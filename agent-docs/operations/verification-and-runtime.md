## Current Command Meaning

- `pnpm build`: executes the solution-style `tsc -b` build across the workspace package graph, with `packages/contracts` library emit ordered through `packages/contracts/tsconfig.build.json`, `packages/hosted-execution` built before `packages/runtime-state` and the hosted app callers, `packages/runtime-state` shared-runtime helpers built before query, device-syncd, inboxd, assistant-runtime, and CLI consumers, and downstream runtime packages, including `packages/parsers`, resolved through project references.
- `pnpm clean`: removes workspace build artifacts such as `dist/`, `.next/`, `.next-smoke/`, `.test-dist/`, and `*.tsbuildinfo`, then prunes untracked generated JS/declaration sidecars that sit next to tracked TypeScript source files under `packages/`, `apps/`, and `e2e/`. Hosted `apps/web` interactive dev now uses `apps/web/.next-dev`, which intentionally stays outside the shared clean/build lane so repo build/test/clean commands do not tear down a running hosted dev server, while smoke output is disposable under `apps/web/.next-smoke`.
- `pnpm no-js`: first prunes untracked generated JS/declaration sidecars that sit next to tracked TypeScript source files, then fails if handwritten `.js`, `.mjs`, `.cjs`, or `.d.ts` source files remain under `packages/`, `apps/`, or `e2e/` outside the explicit Next.js stub allowlist plus the explicit `packages/web/postcss.config.mjs` and `apps/web/postcss.config.mjs` framework config paths, or if tracked `.env` / `.env.*` private files or generated residue such as `dist/`, `.next/`, `.next-dev/`, `.next-smoke/`, `.test-dist/`, or `*.tsbuildinfo` is committed there.
- `pnpm typecheck`: validates shell syntax, syntax-checks the root `.mjs` release helpers, runs the workspace-boundary audit that blocks sibling build-artifact aliases including `.next-dev` and `.next-smoke` plus relative cross-package imports, typechecks repo-owned TS tools, runs each package/app's local no-emit typecheck script, bootstraps the exact Next route-type stubs referenced by `packages/web/next-env.d.ts` and `apps/web/next-env.d.ts`, and verifies the shared `tsc -b` build, including `packages/hosted-execution`, `packages/runtime-state`, `packages/device-syncd`, `packages/inboxd`, `packages/parsers`, `packages/assistant-runtime`, `packages/web`, `apps/web`, and `apps/cloudflare`.
- `pnpm test`: runs docs drift enforcement, repeats the workspace-boundary audit, runs the tracked-artifact hygiene guard, then runs package/app verification for `contracts`/`hosted-execution`/`runtime-state`/`cli`/`core`/`importers`/`device-syncd`/`inboxd`/`parsers`/`assistant-runtime`/`query`/`packages/web`/`apps/web`/`apps/cloudflare`, including the Oura and WHOOP config/provider/service daemon tests, shared hosted execution auth/env/client coverage, hosted verified-email env helper coverage, hosted control-plane auth/session/onboarding/share/Linq-ingress/build coverage plus the durable execution-outbox path for hosted share acceptance, hosted onboarding activation/Linq dispatches, hosted device-sync wakes, and the hosted verified-email sync route, hosted onboarding wallet-forwarding and RevNet webhook idempotency tests, hosted-runner signed-dispatch and one-shot runner coverage including explicit activation bootstrap, separately encrypted per-user env control/persistence handling, hosted verified-email self-target reconciliation, isolated child-process execution, Workers-runtime dispatch/DO-alarm/runner-control coverage, and hosted share acceptance import handling, assistant cron plus assistant status/doctor/stop resilience CLI/runtime tests, failover/outbox/fault-injection robustness coverage, Linq webhook/outbound adapter coverage, AgentMail email connector/delivery/setup discovery tests, and fixture/scenario integrity verification through `e2e/smoke/verify-fixtures.ts`.
- `pnpm test:packages`: runs the tracked-artifact hygiene guard first, then runs executable runtime package/app checks plus the built CLI verification path, executes `packages/web test` (focused web Vitest fixture plus the default Next.js production build), executes `apps/web test` (Prisma generate, hosted auth/session/onboarding/share/Linq-ingress/RevNet Vitest, the hosted cold-boot `next dev` smoke against `apps/web/.next-smoke`, and the hosted default Next.js production build under `apps/web/.next`), executes `apps/cloudflare verify` (focused hosted-runner typecheck once plus the Node and Workers-runtime Vitest lanes), then executes the root Vitest suite with `--no-coverage`, including device-syncd service/runtime tests plus the Oura and WHOOP provider/config tests and parser-layer tests discovered under `packages/parsers/test/**`.
- `pnpm test:packages:coverage`: runs the tracked-artifact hygiene guard first, then runs the same package/app runtime path, including `packages/web test`, `apps/web test`, and `apps/cloudflare verify`, rebuilds the workspace once more so the final repo-wide Vitest pass reads fresh CLI runtime artifacts after the hosted app lanes, then executes the root Vitest suite with deterministic single-file execution and V8 coverage using the sibling CLI thresholds (`perFile`, `lines/functions/statements=85`, `branches=80`) over the targeted `core`, explicitly audited `hosted-execution` auth/client/contracts/env/routes, `importers`, and `query` export-pack/model/search/summaries/timeline source surface, while device-syncd, inbox, parser, assistant-runtime, and hosted-app tests still execute under coverage.

This keeps ignored local artifacts out of the upload bundle without requiring a clean development worktree, while raw clone archives remain unsafe.
- `pnpm test:smoke`: runs only the fixture/scenario integrity verifier.
- `pnpm release:check`: installs with `--frozen-lockfile`, builds the workspace, runs `pnpm verify:repo`, validates the fixed-version monorepo release manifest plus publish metadata, and packs every publishable package into temporary tarballs with `pnpm pack`.

## Incur-Backed CLI Guardrails

- Model nested CLI verbs with real incur router groups. Do not use argv rewrites or synthetic action args to mimic nested commands, because `--schema`, `--llms`, `skills add`, and command-map typegen only stay truthful when the router tree itself is truthful.
- Treat incur-owned transport and discovery features as framework behavior: `--format`, `--json`, `--verbose`, `--schema`, `--llms`, `skills add`, and `--mcp`. Command-surface docs should describe Murph semantics and payloads, not restate incur defaults command-by-command unless the repo is deliberately constraining them.
- Keep `packages/cli/src/index.ts` default-exporting the root CLI and refresh `packages/cli/src/incur.generated.ts` whenever command topology changes. If `incur gen` is blocked by an unrelated build failure, record that explicitly in the handoff instead of silently leaving stale generated types.
- `packages/cli/test/cli-test-helpers.ts` executes `packages/cli/dist/bin.js`, so source checks like `pnpm exec tsx packages/cli/src/bin.ts ...` are only a diagnostic shortcut. Final verification still needs the built CLI path or a clearly documented unrelated blocker.
