## Current Command Meaning

- `pnpm build`: executes the solution-style `tsc -b` build across the workspace package graph, with `packages/contracts` library emit ordered through `packages/contracts/tsconfig.build.json`, `packages/runtime-state` shared-runtime helpers built before query, device-syncd, inboxd, and CLI consumers, and downstream runtime packages, including `packages/parsers`, resolved through project references.
- `pnpm typecheck`: validates shell syntax, syntax-checks the root `.mjs` release helpers, typechecks repo-owned TS tools, runs each package/app’s local no-emit typecheck script, and verifies the shared `tsc -b` build, including `packages/runtime-state`, `packages/device-syncd`, `packages/inboxd`, `packages/parsers`, `packages/web`, and `apps/web`.
- `pnpm test`: runs docs drift enforcement, package/app verification for `contracts`/`runtime-state`/`cli`/`core`/`importers`/`device-syncd`/`inboxd`/`parsers`/`query`/`packages/web`/`apps/web`, including the Oura and WHOOP config/provider/service daemon tests, hosted control-plane auth/crypto/build coverage, assistant cron CLI/runtime tests, Linq webhook/outbound adapter coverage, AgentMail email connector/delivery/setup discovery tests, and fixture/scenario integrity verification through `e2e/smoke/verify-fixtures.ts`.
- `pnpm test:packages`: runs executable runtime package/app checks plus the built CLI verification path, executes `packages/web test` (focused web Vitest fixture plus Next.js webpack build), executes `apps/web test` (Prisma generate, hosted auth/crypto Vitest, and hosted Next.js webpack build), then executes the root Vitest suite with `--no-coverage`, including device-syncd service/runtime tests plus the Oura and WHOOP provider/config tests and parser-layer tests discovered under `packages/parsers/test/**`.
- `pnpm test:packages:coverage`: runs the same package/app runtime path, including `packages/web test` and `apps/web test`, then executes the root Vitest suite with deterministic single-file execution and V8 coverage using the sibling CLI thresholds (`perFile`, `lines/functions/statements=85`, `branches=80`) over the targeted `core`/`importers`/`query` source surface, including `packages/query/src/search-sqlite.ts`, while device-syncd, inbox, parser, and hosted-app tests still execute under coverage.
- `pnpm test:coverage`: runs doc-gardening validation, `pnpm test:packages:coverage`, verifies every documented baseline command has smoke coverage plus a golden-output scaffold, and fails if handwritten `.js`, `.mjs`, `.cjs`, or `.d.ts` source files remain under `packages/`, `apps/`, or `e2e/` aside from the framework-generated `packages/web/next-env.d.ts` and `apps/web/next-env.d.ts` allowlist, or if tracked `dist/`, `.next/`, `.test-dist/`, or `*.tsbuildinfo` residue is committed there.
- `pnpm test:smoke`: runs only the fixture/scenario integrity verifier.
- `pnpm release:check`: installs with `--frozen-lockfile`, builds the workspace, runs `pnpm verify:repo`, validates the fixed-version monorepo release manifest plus publish metadata, and packs every publishable package into temporary tarballs with `pnpm pack`.

## UI Verification Addendum

- For UI-affecting `packages/web` changes, use browser inspection in addition to the scripted checks above.
- Inspect at least one desktop-width and one mobile-width state before handoff.
- Verify no horizontal overflow, clipped controls, broken wrapping, or obvious overlap in the default and changed interaction states.
- Treat the current web test suite as structural/regression coverage, not as proof of visual quality.

## Incur-Backed CLI Guardrails

- Model nested CLI verbs with real incur router groups. Do not use argv rewrites or synthetic action args to mimic nested commands, because `--schema`, `--llms`, `skills add`, and command-map typegen only stay truthful when the router tree itself is truthful.
- Treat incur-owned transport and discovery features as framework behavior: `--format`, `--json`, `--verbose`, `--schema`, `--llms`, `skills add`, and `--mcp`. Command-surface docs should describe Healthy Bob semantics and payloads, not restate incur defaults command-by-command unless the repo is deliberately constraining them.
- Keep `packages/cli/src/index.ts` default-exporting the root CLI and refresh `packages/cli/src/incur.generated.ts` whenever command topology changes. If `incur gen` is blocked by an unrelated build failure, record that explicitly in the handoff instead of silently leaving stale generated types.
- `packages/cli/test/cli-test-helpers.ts` executes `packages/cli/dist/bin.js`, so source checks like `pnpm exec tsx packages/cli/src/bin.ts ...` are only a diagnostic shortcut. Final verification still needs the built CLI path or a clearly documented unrelated blocker.

## Runtime Status

- No deployment target is defined yet.
- Repo-level checks execute canonical write/read paths in `core`, `importers`, `inboxd`, `parsers`, and `query`, build the shared `runtime-state` package, and build the CLI package through the same TypeScript workspace toolchain used for local development.
- Shared runtime-state helpers now own `.runtime` path resolution plus SQLite open defaults for query search, inboxd, and the CLI inbox layer.
- Query-owned lexical search state lives only at `.runtime/search.sqlite`; inbox-owned local state remains at `.runtime/inboxd.sqlite` plus `.runtime/inboxd/*.json`.
- Device sync state lives only at `.runtime/device-syncd.sqlite`; Healthy Bob's CLI-managed daemon launcher state and logs live under `.runtime/device-syncd/`; provider OAuth sessions and encrypted tokens remain outside the canonical vault.
- `vault-cli inbox model bundle|route` can materialize capture-scoped audit artifacts under `derived/inbox/**/assistant/*.json`; those files are rebuildable and non-canonical.
- `vault-cli inbox model route` may call either the AI Gateway or an operator-supplied OpenAI-compatible endpoint. Automated repo checks do not execute live network model calls.
- `vault-cli assistant ask|chat|deliver|run|session|memory|cron` persist or inspect local assistant state under `assistant-state/`, including explicit conversation bindings, timestamps/turn counts, provider session references, local transcript files, inbox-routing and channel auto-reply cursors, enabled auto-reply channels, assistant cron job definitions plus run history under `assistant-state/cron/`, and Markdown assistant memory for naming, response preferences, standing instructions, selected health context, and recent project context; fresh sessions now inject only a small core memory block, recent notes are tool-retrieved on demand through `assistant memory search|get|forget`, assistant cron management is exposed through the same bounded MCP-backed CLI path, assistant-written memory commits are bound to the real active user turn on the host side, health memory is private-context-gated, and provider-native transcript history plus channel-native send history may still stay external when adapters support them. Current outbound channel support covers iMessage, Telegram, Linq, and AgentMail-backed email, and email setup can now reuse a discovered or explicit existing inbox when the API key cannot create new inboxes.
- Provider-backed Codex assistant turns default to `workspace-write` sandboxing plus `on-request` approvals so the CLI-backed MCP memory tool path and routine local commands are available without dropping directly to full-access defaults.
- When the built CLI artifact is present, assistant-memory operations are exposed to Codex as a bounded local MCP server rooted at the active vault/session context; CLI fallback remains available for direct operator use, and the live provider path should use that MCP tool surface rather than a separate localhost bridge.
- Assistant-memory writes now carry minimal provenance metadata (`writtenBy`, `sessionId`, `turnId`) and use a vault-scoped write lock so concurrent assistant processes or automations cannot interleave Markdown commits.
- `vault-cli` and `healthybob` load local `.env.local` first and then `.env` from the launch cwd before command dispatch, while preserving already-exported shell variables as higher precedence. - `vault-cli assistant run` can now run without `--model` when Healthy Bob only needs configured channel auto-reply such as iMessage, Telegram, Linq, or AgentMail email; when a model is supplied it also continues to perform canonical inbox triage, dedicated self-chat flows can opt into self-authored captures plus age-based session rollover, and due assistant cron jobs are processed while that loop stays active for the selected vault.
