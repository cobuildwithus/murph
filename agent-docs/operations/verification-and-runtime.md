## Current Command Meaning

- `pnpm build`: executes the solution-style `tsc -b` build across the workspace package graph, with `packages/contracts` library emit ordered through `packages/contracts/tsconfig.build.json`, `packages/runtime-state` shared-runtime helpers built before query, device-syncd, inboxd, and CLI consumers, and downstream runtime packages, including `packages/parsers`, resolved through project references.
- `pnpm clean`: removes workspace build artifacts such as `dist/`, `.next/`, `.test-dist/`, and `*.tsbuildinfo`, then prunes untracked generated JS/declaration sidecars that sit next to tracked TypeScript source files under `packages/`, `apps/`, and `e2e/`.
- `pnpm no-js`: first prunes untracked generated JS/declaration sidecars that sit next to tracked TypeScript source files, then fails if handwritten `.js`, `.mjs`, `.cjs`, or `.d.ts` source files remain under `packages/`, `apps/`, or `e2e/` outside the explicit Next.js stub allowlist, or if tracked `.env` / `.env.*` private files or generated residue such as `dist/`, `.next/`, `.test-dist/`, or `*.tsbuildinfo` is committed there.
- `pnpm typecheck`: validates shell syntax, syntax-checks the root `.mjs` release helpers, runs the workspace-boundary audit that blocks sibling-`dist` typecheck aliases and relative cross-package imports, typechecks repo-owned TS tools, runs each package/app's local no-emit typecheck script, and verifies the shared `tsc -b` build, including `packages/runtime-state`, `packages/device-syncd`, `packages/inboxd`, `packages/parsers`, `packages/web`, `apps/web`, and `apps/cloudflare`.
- `pnpm test`: runs docs drift enforcement, repeats the workspace-boundary audit, runs the tracked-artifact hygiene guard, then runs package/app verification for `contracts`/`runtime-state`/`cli`/`core`/`importers`/`device-syncd`/`inboxd`/`parsers`/`query`/`packages/web`/`apps/web`/`apps/cloudflare`, including the Oura and WHOOP config/provider/service daemon tests, hosted control-plane auth/session/onboarding/share/Linq-ingress/build coverage, hosted-runner signed-dispatch and one-shot runner coverage including per-user env control/persistence handling and hosted share acceptance import handling, assistant cron plus assistant status/doctor/stop resilience CLI/runtime tests, failover/outbox/fault-injection robustness coverage, Linq webhook/outbound adapter coverage, AgentMail email connector/delivery/setup discovery tests, and fixture/scenario integrity verification through `e2e/smoke/verify-fixtures.ts`.
- `pnpm test:packages`: runs the tracked-artifact hygiene guard first, then runs executable runtime package/app checks plus the built CLI verification path, executes `packages/web test` (focused web Vitest fixture plus Next.js webpack build), executes `apps/web test` (Prisma generate, hosted auth/session/onboarding/share/Linq-ingress Vitest, and hosted Next.js webpack build), executes `apps/cloudflare test` (focused hosted-runner typecheck plus Vitest), then executes the root Vitest suite with `--no-coverage`, including device-syncd service/runtime tests plus the Oura and WHOOP provider/config tests and parser-layer tests discovered under `packages/parsers/test/**`.
- `pnpm test:packages:coverage`: runs the tracked-artifact hygiene guard first, then runs the same package/app runtime path, including `packages/web test`, `apps/web test`, and `apps/cloudflare test`, then executes the root Vitest suite with deterministic single-file execution and V8 coverage using the sibling CLI thresholds (`perFile`, `lines/functions/statements=85`, `branches=80`) over the targeted `core`/`importers`/`query` source surface, including `packages/query/src/search-sqlite.ts`, while device-syncd, inbox, parser, and hosted-app tests still execute under coverage.
- `pnpm test:coverage`: runs doc-gardening validation, `pnpm test:packages:coverage`, verifies every documented baseline command has smoke coverage plus a golden-output scaffold, and fails if handwritten `.js`, `.mjs`, `.cjs`, or `.d.ts` source files remain under `packages/`, `apps/`, or `e2e/` aside from the framework-generated `packages/web/next-env.d.ts` and `apps/web/next-env.d.ts` allowlist, or if tracked `.env` / `.env.*` private files or generated residue such as `dist/`, `.next/`, `.test-dist/`, or `*.tsbuildinfo` is committed there. Untracked generated sidecars adjacent to tracked TypeScript source files are pruned before that guard runs. Tracked `.env.example` files remain allowed.
- `pnpm zip:src` and `scripts/package-audit-context.sh`: shell through `pnpm no-js`, which first prunes untracked generated JS/declaration sidecars that sit next to tracked TypeScript source files and then runs the tracked-artifact hygiene guard, before building the source/review bundle from git-visible files while filtering blocked local residue such as `.env` / `.env.*`, `dist/`, `.next/`, `.test-dist/`, and `*.tsbuildinfo` paths out of the manifest. This keeps ignored local artifacts out of the upload bundle without requiring a clean development worktree, while raw clone archives remain unsafe.
- `pnpm test:smoke`: runs only the fixture/scenario integrity verifier.
- `pnpm release:check`: installs with `--frozen-lockfile`, builds the workspace, runs `pnpm verify:repo`, validates the fixed-version monorepo release manifest plus publish metadata, and packs every publishable package into temporary tarballs with `pnpm pack`.

## UI Verification Addendum

- For UI-affecting `packages/web` changes, use browser inspection in addition to the scripted checks above.
- Inspect at least one desktop-width and one mobile-width state before handoff.
- Verify no horizontal overflow, clipped controls, broken wrapping, or obvious overlap in the default and changed interaction states.
- Treat the current web test suite as structural/regression coverage, not as proof of visual quality.

## Incur-Backed CLI Guardrails

- Model nested CLI verbs with real incur router groups. Do not use argv rewrites or synthetic action args to mimic nested commands, because `--schema`, `--llms`, `skills add`, and command-map typegen only stay truthful when the router tree itself is truthful.
- Treat incur-owned transport and discovery features as framework behavior: `--format`, `--json`, `--verbose`, `--schema`, `--llms`, `skills add`, and `--mcp`. Command-surface docs should describe Healthy Bob semantics and payloads, not restate incur defaults command-by-command unless the repo is deliberately constraining them.
- Keep `packages/cli/src/index.ts` default-exporting the root CLI and refresh `packages/cli/src/incur.generated.ts` whenever command topology changes. If `incur gen` is blocked by an unrelated build failure, record that explicitly in the handoff instead of silently leaving stale generated types.
- `packages/cli/test/cli-test-helpers.ts` executes `packages/cli/dist/bin.js`, so source checks like `pnpm exec tsx packages/cli/src/bin.ts ...` are only a diagnostic shortcut. Final verification still needs the built CLI path or a clearly documented unrelated blocker.
