## Scoped Verification Mode

The text-only docs/process fast path above is not a scoped-verification exception. It is the default verification rule for eligible Markdown-only docs work.

Scoped verification may replace the repo-wide baseline only when all of the following are true:

1. The change is narrow and bounded to one subsystem or one docs/process lane rather than a broad refactor.
2. `pnpm typecheck` or `pnpm test:coverage` are already credibly known red for unrelated reasons in the current branch or working session.
3. You can name the exact failing command and failing target, and explain why your diff did not cause that failure.
4. You run the highest-signal scoped checks available for the touched surface and record the evidence in handoff.

Scoped verification is allowed for narrow changes such as:

- low-risk repo-internal workflow/tooling changes where `pnpm typecheck` plus direct touched-file checks fully exercise the changed surface
- docs/process-only updates outside the text-only Markdown fast path when repo-wide checks are already known red and manual readback confirms the touched docs are internally consistent
- package-local or app-local fixes with a focused test, typecheck, verify, or scenario command that exercises the changed surface directly; for agent/local iteration on repo code, prefer `pnpm test:diff` first so the scope expands from changed owners to their workspace dependents automatically
- small config changes with a direct validation command or targeted test covering the changed contract

Scoped verification is not allowed when the change is broad, cross-cutting, or high-risk, including schema/storage changes, billing/auth/trust-boundary changes, deploy/runtime entrypoint changes, or refactors that touch multiple subsystems. Those changes still need the full repo-wide baseline unless the user explicitly says otherwise.

When using scoped verification, handoff must include:

- that scoped verification mode was used
- which repo-wide commands were omitted or left red
- the prior unrelated failing command(s) and target(s)
- the focused commands or direct scenario checks that were run instead

## Pnpm Guard

- Do not add `--config.verify-deps-before-run=false` to repo verification commands or package-local verification commands.
- If `pnpm` reports `ERR_PNPM_VERIFY_DEPS_BEFORE_RUN`, treat that as a real workspace-state problem to fix or report, not a guard to bypass.
- Verification evidence in plans, prompts, and handoff should use the normal command forms unless a user explicitly asks for a different command shape.

## Low-Risk Fast Path

Use the low-risk repo-internal workflow/tooling fast path when all of the following are true:

1. The diff stays within repo-internal docs/process/verification tooling paths such as `agent-docs/**`, `docs/**`, `scripts/**`, `AGENTS.md`, `ARCHITECTURE.md`, `README.md`, `vitest.config.ts`, or root `tsconfig*.json`.
2. The change does not touch app/package runtime behavior, product behavior, persisted-state logic, auth/trust boundaries, or deploy surfaces.
3. `pnpm typecheck` plus direct touched-file checks are enough to exercise the changed surface directly.

When that fast path applies:

- `pnpm typecheck` remains required.
- Direct checks on the touched tooling files remain required.
- `pnpm test`, `pnpm test:coverage`, and `pnpm verify:acceptance` are optional and should be skipped unless the touched files really need broader proof.

## Current Command Meaning

- `pnpm build`: executes the solution-style `tsc -b` build across the workspace package graph, with `packages/contracts` library emit ordered through `packages/contracts/tsconfig.build.json`, `packages/hosted-execution` built before `packages/runtime-state` and the hosted app callers, `packages/runtime-state` shared-runtime helpers built before query, device-syncd, inboxd, assistant-runtime, and CLI consumers, the CLI built before `packages/assistantd` so the daemon can consume the published `@murphai/murph` surface, and downstream runtime packages, including `packages/parsers`, resolved through project references.
- `pnpm build:workspace:clean`: clears the referenced workspace-build runtime-package outputs plus their project-reference `tsbuildinfo` files first, then runs the root TypeScript project-reference graph through one `tsc -b tsconfig.json` invocation. Use this when the build itself needs clean-build semantics, such as release or CI proof.
- `pnpm build:workspace:incremental`: runs that same root TypeScript project-reference graph without first deleting outputs or incremental metadata, so warm local runs can reuse the package-local `.tsbuildinfo` files now kept outside `dist/`.
- `pnpm build:test-runtime`: clears the narrower runtime-package `dist/` outputs first, then runs the focused TypeScript project-reference graph in `tsconfig.test-runtime.json`, building only the workspace runtime artifacts needed for built CLI consumers and the root package Vitest lane.
- `pnpm build:test-runtime:prepared`: reruns that same focused TypeScript project-reference graph without first deleting the existing runtime-package outputs, so repeated local CLI verification can reuse already-prepared runtime artifacts while still refreshing changed files. It now tries the incremental pass first, falls back to a forced rebuild only when the post-build smoke check sees missing artifacts, and verifies the reusable `packages/cli/dist/cli-entry.js` module alongside the existing built CLI runtime outputs.
- `pnpm build`: runs package-local `build` scripts through pnpm's topologically sorted workspace runner with `MURPH_BUILD_WORKSPACE_CONCURRENCY` defaulting to `4`, so package-specific build hooks still run while sibling packages can build in parallel when their dependency edges allow it.
- `pnpm clean`: removes workspace build artifacts such as `dist/`, `.next/`, `.next-smoke/`, `.test-dist/`, and `*.tsbuildinfo` through the repo-owned `scripts/rm-paths.mjs` helper instead of the third-party `rimraf` package, then prunes untracked generated JS/declaration sidecars that sit next to tracked TypeScript source files under `packages/`, `apps/`, and `e2e/`. Hosted `apps/web` interactive dev now uses `apps/web/.next-dev`, which intentionally stays outside the shared clean/build lane so repo build/test/clean commands do not tear down a running hosted dev server, while smoke output is disposable under `apps/web/.next-smoke`.
- `pnpm no-js`: first prunes untracked generated JS/declaration sidecars that sit next to tracked TypeScript source files, then fails if handwritten `.js`, `.mjs`, `.cjs`, or `.d.ts` source files remain under `packages/`, `apps/`, or `e2e/` outside the explicit Next.js stub allowlist plus the explicit `apps/web/{postcss,eslint}.config.mjs` framework config paths, or if tracked `.env` / `.env.*` private files or generated residue such as `dist/`, `.next/`, `.next-dev/`, `.next-smoke/`, `.test-dist/`, or `*.tsbuildinfo` is committed there.
- `pnpm deps:guard`: validates the repo dependency policy by requiring the committed `pnpm-lock.yaml`, a pinned root `packageManager`, matching `engines.pnpm`, the root pnpm supply-chain settings in `pnpm-workspace.yaml`, `workspace:` protocol for internal packages, and registry-sourced third-party specs instead of git/url/file/alias shortcuts.
- `pnpm deps:audit`: runs `pnpm audit --audit-level=high` so dependency changes can be screened for known high-severity advisories before handoff.
- `pnpm deps:ignored-builds`: shows dependency install scripts that pnpm blocked so dependency updates can be reviewed instead of silently executing new lifecycle code.
- `pnpm deps:approve-builds`: records reviewed install-script approvals into `pnpm-workspace.yaml` after a trusted-machine dependency refresh.
- `pnpm typecheck`: validates shell syntax, syntax-checks the root `.mjs` release helpers, runs `pnpm deps:guard` to require the committed lockfile plus the repo dependency-source policy, runs the workspace-boundary audit that blocks sibling build-artifact aliases including `.next-dev` and `.next-smoke` plus relative cross-package imports, runs the workspace package dependency cycle guard, typechecks repo-owned TS tools, bootstraps the exact Next route-type stub referenced by `apps/web/next-env.d.ts`, builds `packages/contracts`, then runs each package/app's local no-emit `typecheck` script through pnpm's topologically sorted workspace runner with bounded parallelism. It no longer runs the full workspace `tsc -b` emit build by default, so the local loop does not pay for both solution-build proof and package/app no-emit proof on every run. The package-local no-emit configs still keep explicit `tsbuildinfo` files so repeated local runs can reuse incremental state. That final package/app pass defaults to `MURPH_TYPECHECK_WORKSPACE_CONCURRENCY=4` locally and `2` in CI, while `MURPH_TYPECHECK_WORKSPACE_CONCURRENCY=1` restores the old fully serial package loop. The root verify harness now also prints per-step timing so agents can see where time is actually going.
- `pnpm test:diff`: the default agent/local iteration lane for repo code when you want more signal than one package-local command without paying for full repo acceptance. It analyzes the current worktree diff against `HEAD` plus untracked files, maps changed files to workspace owners, expands to reverse workspace dependents, then runs the matching package/app-local `typecheck`, `test`, `verify`, and built-CLI verification steps for that affected subset. When the changed files stay inside the repo-internal docs/process/verification-tooling fast path, or otherwise stay on non-workspace owner files only, it stays on the narrow tooling checks (`shell syntax`, `node --check`, repo TS tools typecheck, and dependency policy) instead of fanning out to app/package lanes or running unrelated workspace-boundary audits; when root workspace-manifest files such as `pnpm-lock.yaml` or `pnpm-workspace.yaml` change, it broadens to the whole workspace. Use `bash scripts/workspace-verify.sh test:diff <path ...>` on dirty branches when you need to scope the lane to the files for the current task instead of every unrelated local edit.
- `pnpm test`: runs only the fast deterministic local behavior loop: `packages/contracts test`, `packages/openclaw-plugin test`, the prepared-runtime CLI/package Vitest lane, and fixture/scenario-manifest verification without coverage. It no longer runs dependency-policy or workspace-boundary guards, tracked-artifact hygiene, app verification, or any git/worktree-sensitive skip heuristics. Local runs still overlap repo Vitest with scenario-manifest verification when `MURPH_TEST_LANES_PARALLEL` allows it; CI stays sequential by default. Command retries remain opt-in through `MURPH_VERIFY_RETRY_COUNT`, with a default of `0` so red local loops fail fast.
- `pnpm docs:drift`: runs the manual durable-doc drift check. Use it when a task intentionally changes `agent-docs/**`, `ARCHITECTURE.md`, or other durable repo docs and you want the old index/truthfulness guard explicitly, without making every default `pnpm test` run sensitive to unrelated dirty-tree doc work.
- `pnpm test:packages`: runs the tracked-artifact hygiene guard first, runs `packages/contracts test`, performs the prepared CLI runtime graph plus CLI package-shape verification for built-runtime consumers, then executes the root multi-project Vitest suite through `vitest.config.ts` with `--no-coverage`. That root package lane now runs the full package-local test surfaces for every repo-owned Vitest package wired into `vitest.config.ts`, including `inbox-services`, `inboxd-imessage`, and `vault-usecases`, and the CLI portion now runs the same full six serial-safe workspace buckets used by the package-local CLI suite instead of a narrower hand-picked subset. It still intentionally leaves the heavier web/app verify work to `pnpm test:apps`. `MURPH_VITEST_MAX_WORKERS` still defaults to `75%` locally or `50%` in CI when the environment does not override it, file-level Vitest parallelism stays enabled locally by default, in-file suite concurrency remains opt-in unless `MURPH_VITEST_SUITE_CONCURRENCY` explicitly enables it, `MURPH_VITEST_FILE_PARALLELISM` can force file parallelism, and `MURPH_VITEST_MAX_CONCURRENCY` / `MURPH_CLI_VITEST_MAX_CONCURRENCY` cap concurrent tests within a file when suite concurrency is enabled. Hosted web and Cloudflare Node lanes still use their dedicated workspace configs outside this command.
- `pnpm test:apps`: runs the dedicated heavier app verification lane by executing `apps/web verify` and `apps/cloudflare verify`. Local runs default to parallel execution and CI defaults to sequential execution unless `MURPH_APP_VERIFY_PARALLEL` overrides the behavior. Inside that lane, the app-local `verify` wrappers also default to overlapping their own heavy substeps locally unless `MURPH_VERIFY_STEP_PARALLEL` overrides them: `apps/web` runs `prisma:generate`, then overlaps both `pnpm test` and `pnpm lint` with the heavier smoke/build work while keeping `pnpm dev:smoke` and `next build` sequential so Next does not detect a competing build, and `apps/cloudflare` overlaps `test:node` with `test:workers` after `typecheck`. The hosted-web lane intentionally relies on the `next build` TypeScript validation it already performs instead of paying for a separate pre-build `typecheck:prepared` pass in that same wrapper. The tag-driven release workflow now opts into both CI overrides explicitly so release verification does not serialize the whole app lane. The hosted-web and Cloudflare Node Vitest lanes underneath those verify commands now use split workspace buckets with app-local worker caps defaulting to `MURPH_APP_VITEST_MAX_WORKERS`, then `MURPH_VITEST_MAX_WORKERS`, then `50%` locally or `25%` in CI. Those app lanes also enable file-level Vitest parallelism locally by default, disable it in CI by default, and accept `MURPH_VITEST_FILE_PARALLELISM` as explicit overrides.
- `pnpm test:packages:coverage`: runs the tracked-artifact hygiene guard first, performs the prepared CLI runtime graph plus CLI package-shape verification for built-runtime consumers, then enforces coverage by running every package-local `test:coverage` command under `packages/*`. That package coverage lane now covers `assistant-cli`, `assistant-engine`, `assistant-runtime`, `assistantd`, `cli`, `cloudflare-hosted-control`, `contracts`, `core`, `device-syncd`, `gateway-core`, `gateway-local`, `hosted-execution`, `importers`, `inbox-services`, `inboxd`, `inboxd-imessage`, `messaging-ingress`, `openclaw-plugin`, `operator-config`, `parsers`, `query`, `runtime-state`, `setup-cli`, and `vault-usecases`, all on the same canonical repo thresholds. The root package-coverage lane bounds outer package fanout to `MURPH_PACKAGE_COVERAGE_CONCURRENCY=4` locally and `1` in CI by default instead of launching every package at once, and also nudges package-local coverage runs to `MURPH_PACKAGE_COVERAGE_VITEST_MAX_WORKERS=100%` locally and `50%` in CI by default through `MURPH_VITEST_MAX_WORKERS` for that lane. `MURPH_PACKAGE_COVERAGE_CONCURRENCY` and `MURPH_PACKAGE_COVERAGE_VITEST_MAX_WORKERS` can widen or narrow those package-coverage defaults when needed. When one package coverage job fails inside a parallel batch, the root harness now lets the other in-flight package jobs in that batch finish and then reports the failed package labels before exiting non-zero. It still leaves the heavier web/app verify work to `pnpm test:apps`. Package-local `test` commands remain the full per-package loop and are not aliases for this coverage lane.
- `pnpm test:coverage`: runs the explicit coverage-focused acceptance lane. It repeats `pnpm deps:guard`, repeats the workspace-boundary audit, runs the workspace package dependency cycle guard, runs doc-gardening validation, then on local machines defaults to overlapping `pnpm test:packages:coverage`, `pnpm test:apps`, and scenario-manifest coverage verification through `e2e/smoke/verify-fixtures.ts --coverage`; CI stays sequential unless `MURPH_TEST_LANES_PARALLEL` overrides it. Untracked generated sidecars adjacent to tracked TypeScript source files are pruned before that guard runs. Tracked `.env.example` files remain allowed.
- `pnpm verify:acceptance`: the canonical repo acceptance gate. It runs `pnpm typecheck` first, then `pnpm test:coverage`, so CI/release callers have one explicit command for the full static-plus-acceptance contract.
- `pnpm zip:src` and `scripts/package-audit-context.sh`: shell through `pnpm no-js`, which first prunes untracked generated JS/declaration sidecars that sit next to tracked TypeScript source files and then runs the tracked-artifact hygiene guard, before building the source/review bundle from git-visible files while scanning `config/**` alongside app/package code and filtering blocked local residue such as `.env` / `.env.*`, `dist/`, `.next/`, `.next-dev/`, `.next-smoke/`, `.test-dist/`, and `*.tsbuildinfo` paths out of the manifest. Treat it as the release-specific extension of `pnpm verify:acceptance`, with the extra clean-build proof layered on top.

## Incur-Backed CLI Guardrails

- Model nested CLI verbs with real incur router groups. Do not use argv rewrites or synthetic action args to mimic nested commands, because `--schema`, `--llms`, `skills add`, and command-map typegen only stay truthful when the router tree itself is truthful.
