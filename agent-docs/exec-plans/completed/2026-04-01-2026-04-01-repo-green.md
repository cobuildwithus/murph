## Goal

- Restore required repo verification to green on the current worktree without overwriting unrelated in-flight changes.

## Success criteria

- `pnpm typecheck` passes.
- `pnpm test` passes.
- `pnpm test:coverage` passes.
- `pnpm --dir apps/web lint` passes.
- The current dirty worktree stays limited to intentional repo-green fixes plus the existing hosted-web batch.

## Scope

- In scope:
- Root verification failures exposed by the current worktree.
- The new hosted-web `dev-local` test helper fix already landed while verifying the batch.
- Minimal code, test, or config changes in touched packages needed to make the required checks pass.
- Out of scope:
- Unrelated warning cleanup that does not block required verification.
- Broad refactors or behavioral redesign outside the failing verification paths.

## Constraints

- Technical constraints:
- Preserve unrelated dirty-tree edits and active overlapping lanes.
- Do not revert or rewrite the existing hosted-web UI/onboarding batch while fixing repo-level blockers.
- Product/process constraints:
- Keep the coordination ledger current while this lane is active.
- Commit only after the repo is green or after any remaining failure is proven unrelated.

## Risks and mitigations

1. Risk: Verification blockers may live in packages outside the current dirty batch.
   Mitigation: Fix only the minimum needed, rerun the failing commands, and avoid widening into speculative cleanup.
2. Risk: Active overlapping work in `apps/web`, `packages/assistant-core`, or `packages/device-syncd` could be clobbered.
   Mitigation: Read current file state before edits and keep changes narrowly targeted to verification failures.

## Tasks

1. Capture the failing required verification output and isolate the first real blocker.
2. Fix the blocker with the smallest safe code change.
3. Re-run the affected verification command until it passes.
4. Run the remaining required verification commands and fix any additional blockers.
5. Once green, close the plan and prepare the scoped commit for the full dirty batch.

## Decisions

- Treat repo-green as the priority over immediate commit because the user redirected the task.
- Use a dedicated plan because the work is now multi-step and may span multiple packages.
