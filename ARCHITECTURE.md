# Murph Architecture

Last verified: 2026-03-30

## Module Map

- `packages/contracts`: canonical Zod contracts, parse helpers, TypeScript types, and generated JSON Schema artifacts
- `packages/hosted-execution`: shared hosted control-plane contracts, HMAC signing/verification helpers, env readers, route builders, side-effect codecs, and typed clients for the hosted web and Cloudflare execution apps
- `packages/runtime-state`: shared `.runtime` path resolution plus rebuildable local-state path/default helpers used by query, inboxd, the local assistant runtime, and hosted bundle helpers
- `packages/core`: the only package allowed to mutate canonical vault data
- `packages/importers`: ingestion adapters that parse external files or provider API snapshots, normalize them behind registry-based adapters, and delegate all writes to core
- `packages/device-syncd`: published local device OAuth/webhook/reconcile runtime with an authenticated localhost control plane, optional separate public callback/webhook ingress, a reusable shared public-ingress core for future hosted/tunneled callback surfaces, the canonical `@murph/device-syncd/client` control-plane client/contracts surface for non-daemon callers, encrypted provider credentials outside the vault, and normalized provider snapshot imports through importers/core
- `packages/inboxd`: inbox capture ingestion/runtime package that persists canonical raw inbox evidence plus a first-class append-only inbox-capture ledger while keeping inbox-only cursors, source-specific checkpoints, capture indexes, and attachment job state in local SQLite state; current first-class connectors cover iMessage, Telegram, Linq webhook delivery, and AgentMail-backed email
- `packages/parsers`: local-first attachment parsing, parser-service helpers, and derived artifact publication under `derived/inbox/**`
- `packages/query`: read helpers, export-pack generation, and the optional lexical search index over canonical vault data
- `packages/assistantd`: published local assistant daemon package with a bearer-authenticated loopback-only control plane bound to one vault; it fronts steady-state local assistant session/message/status/automation entrypoints through the explicit `@murph/assistant-core` headless surface and now hosts the full local gateway conversation/message/attachment/send/event/permission API over `@murph/gateway-core`, serving reads from the rebuildable local gateway serving store without moving canonical writes out of the CLI/core boundary
- `packages/assistant-runtime`: headless hosted assistant execution surface that exposes one-shot inbox/bootstrap/assistant/outbox/device-sync/share-import runtime behavior behind explicit runtime context, consumes the same `@murph/assistant-core` headless boundary directly instead of a second wrapper package, and no longer re-exports shared hosted-execution contracts that belong to `@murph/hosted-execution`
- `packages/local-web`: local-only Next.js observability app that reads the selected vault on the server through the query package, preferring explicit `VAULT` selection and otherwise falling back to the saved CLI default vault, and may initiate device-auth control-plane actions against `packages/device-syncd`
- `apps/web`: hosted Next.js integration control plane for Vercel-style deployments, backed by Postgres/Prisma for device OAuth sessions, encrypted token escrow, ownership mapping, webhook traces, sparse sync signals, internal snapshot/apply routes for hosted runner hydration, a durable `execution_outbox` for Cloudflare-bound hosted execution intents, immutable hosted AI usage rows imported after successful hosted commits, Linq recipient bindings, queued Linq webhook events, queued hosted Stripe event facts, hosted onboarding webhook receipts for receipt-local side effects, Privy-backed hosted onboarding state, hosted checkout attempts, optional invoice-idempotent RevNet issuance records, one-time encrypted hosted share links, and local-agent pairing while keeping canonical health and inbox data out of the hosted tier
- `apps/cloudflare`: hosted execution plane for signed internal dispatch, per-user coordination via container-enabled Durable Objects, encrypted hosted workspace snapshot storage plus separately encrypted artifact objects, a native Cloudflare container image, a hot per-user gateway projection plus short-retained gateway event log for internal control-plane reads, and one-shot inbox/parser/assistant/device-sync/share-import execution that materializes a temporary local workspace through `packages/assistant-runtime` rather than importing hosted seams from the operator CLI surface
- `packages/cli`: the published `murph` package plus `vault-cli`, an incur-backed typed operator surface over core/importers/query/inboxd plus parser-toolchain queue controls, quick workout capture atop canonical `activity_session` events, inbox model-routing helpers, provider-backed assistant session orchestration, the transport-neutral `@murph/gateway-core` headless conversation/message/event/send contract now shared directly by assistantd and hosted Cloudflare gateway adapters (with MCP still intentionally deferred), a local gateway projection store under `.runtime/gateway.sqlite` now owned by `@murph/gateway-core`, incrementally syncing inbox captures through inboxd's durable local capture mutation cursor while still syncing assistant sessions/outbox via their existing signatures, persisting explicit snapshot bootstrap metadata so empty projections do not rebuild forever, and serving conversation/message/attachment reads plus event polling from the rebuilt store, optional env-routed client access to `packages/assistantd` for steady-state local assistant open/send/update flows plus session/status/outbox/cron inspection and serializable automation control, CLI-owned command/UI/client helpers for the selected vault, out-of-vault assistant memory docs plus one shared bound assistant/vault tool catalog with turn-scoped profiles for assistant memory/state/cron, saved self-target inspection, bounded vault text reads, and vault query/write operations across both Codex-backed and OpenAI-compatible provider turns, outbound iMessage/Telegram/Linq/AgentMail email channel adapters, an onboarding/setup wizard that can reuse or discover existing AgentMail inboxes before provisioning, and local host setup commands for macOS and Linux while keeping iMessage itself macOS-only. The dedicated `@murph/assistant-core` package owns the local-only assistant/inbox/vault/operator-config surface plus the shared `usecases/**`, `inbox-app/**`, and setup/runtime-helper layers for non-CLI consumers and for CLI headless imports; `packages/cli` now imports that owner package directly and keeps only the CLI command/setup glue and daemon-aware wrapper modules that genuinely live at the CLI transport boundary.
- `fixtures/` and `e2e/`: deterministic fixture corpus and end-to-end smoke flows

## Trust Boundaries

- Canonical vault storage is file-native under the vault root.
- Human-facing truth lives in Markdown documents such as `CORE.md`, journal pages, and experiment pages.
- Machine-facing truth lives in append-only JSONL ledgers for events, samples, and audit records.
- Raw imported artifacts are immutable once copied into `raw/`, including normalized device/provider API snapshots under `raw/integrations/**`.
- Parser outputs under `derived/inbox/**` are rebuildable and never canonical health facts.
- Inbox model-routing bundles, plans, and execution results under `derived/inbox/**/assistant/*.json` are rebuildable audit artifacts and never canonical health facts.
- Inbox runtime state is local-only under `.runtime/inboxd.sqlite` plus `.runtime/inboxd/*.json` and is rebuildable from canonical vault evidence under `raw/inbox/**`.
- AgentMail inbox ids, outbound email thread bindings, and channel credentials remain non-canonical local/runtime concerns; AgentMail API keys stay in operator environment variables and never belong in the vault.
- Query search runtime state is local-only under `.runtime/search.sqlite` and is rebuildable from canonical vault evidence.
- Device sync runtime state is local-only under `.runtime/device-syncd.sqlite`, and Murph's daemon launcher state/logs live under `.runtime/device-syncd/`; encrypted provider tokens, OAuth sessions, and webhook/reconcile cursors never belong in the canonical vault. Hosted device-sync control-plane state for `apps/web` belongs in Postgres only and must remain limited to integration/auth state rather than canonical health payloads. Hosted onboarding billing rows, checkout attempts, queued Stripe facts, webhook receipts, the shared `execution_outbox`, immutable hosted AI usage rows, and optional RevNet issuance rows in Postgres are operational/idempotency state only, not canonical health truth.
- Local assistant runtime state is non-canonical under sibling `assistant-state/**`, including sessions, transcripts, outbox/receipt artifacts, diagnostics, status, memory/state documents, and append-only transcript distillations. Those distillations are continuity aids only and must never be treated as canonical health memory or vault truth.
- The gateway plane is a derived operational model over inbox captures, assistant bindings, sent outbox deliveries, and approval state. Local daemon APIs and hosted Durable Objects may materialize hot gateway projections and short-retained event logs for transport-facing reads, but those projections are never canonical health truth and must remain rebuildable from canonical vault evidence plus non-canonical runtime state. The local projection now persists serving tables in `.runtime/gateway.sqlite`, is owned by `@murph/gateway-core`, consumes inboxd's durable local capture mutation cursor for incremental capture sync, persists explicit snapshot bootstrap metadata so empty projections stay incremental instead of rebuilding forever, and keeps full rebuild behavior as the bootstrap or recovery path.
- Hosted execution state for `apps/cloudflare` may store one encrypted hosted workspace snapshot in the existing `vault` bundle slot plus separately encrypted artifact objects and per-user coordination metadata. That hosted workspace snapshot now includes canonical `vault/**`, durable `vault/.runtime/**`, sibling `assistant-state/**`, and the minimal operator-home config needed for explicit `member.activated` bootstrap, while large raw artifacts under `vault/raw/**` are externalized into their own encrypted objects. The runner restores inline workspace files first, materializes only the externalized artifact paths the current run actually needs, preserves untouched artifact refs across later snapshots so old media is not re-downloaded just to survive the next commit, and best-effort deletes only per-user artifact objects no longer referenced by the latest workspace snapshot. Hosted per-user env overrides still live in their own encrypted object keyed independently from the workspace snapshot. Those hosted bundles still represent canonical vault material and must be treated as sensitive data rather than a separate hosted source of truth. The worker-facing control routes are internal only, the per-user Durable Object now keeps its queue/process state in SQLite tables plus encrypted bundle refs rather than one serialized record blob, starts its native Cloudflare container and talks to the runner over the container TCP port, commits only small bundle-ref/version state changes inside Durable Object storage after any R2 I/O completes, persists committed execution journal records with the owning `userId` so recovery can still rebind the Durable Object after partial state loss, and also maintains a hot per-user gateway projection snapshot plus short-retained gateway event log derived from commit/finalize callbacks rather than from a second hosted source of truth. The containerized runner now hydrates hosted device-sync runtime state from `apps/web`, resolves hosted share payload reads and hosted AI usage imports only through worker-owned outbound proxy hosts that inject the bound `userId`, keeps the broad web internal/control token out of the runner environment, launches each hosted job in a scrubbed child env with per-run temp and cache roots instead of inheriting the full supervisor env, avoids the supervisor `HOME` at child launch while still restoring the persisted operator-home snapshot inside the workspace for runtime config, strips ambient supervisor proxy credentials from the isolated child env, reaps the whole child process group when the container request times out or disconnects, persists due hosted side effects on the committed journal, authenticates the outer container invoke/destroy wrapper with the runner control token header before any body is trusted, accepts hosted gateway send requests only through the same route-bound gateway seam, and calls the worker's internal commit/finalize/side-effect routes so post-commit retries can resume outward delivery without rerunning compute. Exact consumed-event tombstones now block replayed hosted dispatch event ids for 30 days; there is no secondary replay filter beyond that authoritative window. Hosted ciphertext envelopes still decrypt by envelope `keyId` through the configured keyring so operators can stage bundle-key rotation without eagerly rewriting older objects. Writes still use one active bundle key at a time, and missing keyring entries still fail closed. This lane does not claim operator-blind privacy or TEE isolation.
- The local web surface is for local operator use and must not expose raw vault paths, home-directory paths, or canonical write capabilities in its rendered payloads. It may initiate device auth and account-control requests only through the local `device-syncd` control plane and its bearer-token contract, whether that daemon was started manually or by Murph's CLI-managed launcher. Its launcher must block framework `.env*` reads.
- Any inbox-to-canonical promotion idempotency must be stored in or derivable from canonical vault evidence, not `.runtime/` alone.
- General assistant/session state belongs outside the canonical vault under `assistant-state/`, including local transcript files, assistant cron scheduler state and run history under `assistant-state/cron/`, per-turn decision receipts under `assistant-state/receipts/`, replay-safe outbound intent journals under `assistant-state/outbox/`, pending provider-usage records under `assistant-state/usage/pending/`, diagnostics events plus snapshot counters under `assistant-state/diagnostics/`, persisted assistant status snapshots under `assistant-state/status.json`, provider failover cooldown state under `assistant-state/failover.json`, generic small JSON scratchpad documents under `assistant-state/state/`, plus non-canonical Markdown memory for naming, response preferences, standing instructions, selected health context, and recent project context; only capture-scoped rebuildable audit artifacts belong under `derived/inbox/**`.
- Provider transcript history and channel-native delivery history should stay with upstream adapters when possible; Murph stores local assistant transcript copies, minimal manual aliases, explicit conversation bindings, automation cursors, enabled auto-reply channel state, timestamps/turn counts, provider session references, cron job definitions and run history, compact system-emitted turn receipts, idempotent outbound intent state, diagnostics counters/warnings, persisted failover cooldown metadata, persisted status snapshots, generic assistant scratchpad documents, and Markdown assistant memory under `assistant-state/`. Assistant-state directories must stay private (`0700`) and assistant-state files must stay private (`0600`). Secret-bearing provider headers for persisted sessions and provider route recovery records live only in private sidecars under `assistant-state/secrets/**`; the general session/recovery JSON keeps only public headers, diagnostics/runtime-event writes redact inline secret material before persistence, and `assistant doctor --repair` can migrate legacy inline headers and repair permissive assistant-state modes in place. Fresh sessions inject only a small core block by default, recent/project memory is retrieved on demand through typed `assistant memory search|get|forget` calls, provider turns share one canonical local assistant/vault tool catalog, but each turn binds only the smallest needed profile: manual ask/deliver and assistant-cron turns can reach the full assistant-state, assistant-memory, assistant-cron, self-target, and vault query/write surface, while assistant auto-reply is narrowed to bounded vault-relative text inspection such as `vault.fs.readText` and must run only on bound-tool-capable provider routes so untrusted inbound content cannot search broader memory, inspect saved outbound routes, or trigger local mutations through Codex direct-CLI fallback. Generic scratchpads are exposed through `assistant state list|show|put|patch|delete`, assistant-written memory is committed only through host-bound writes that attach session/turn provenance, and canonical vault records remain authoritative on conflicts.
- `vault-cli inbox model route` may send a normalized inbox bundle, with raw routing images attached when eligible, to either the AI Gateway or an operator-specified OpenAI-compatible endpoint.

## Control Flow

1. Operators, automations, and future agent layers call `vault-cli` or package APIs.
2. CLI commands stay thin, validate input, and delegate to internal CLI use-case modules that coordinate `packages/core`, `packages/importers`, `packages/query`, `packages/inboxd`, and parser-toolchain helpers from `packages/parsers`. Canonical mutation flows for experiments, journal pages, providers, events, vault summary updates, and inbox journal/experiment-note promotions must route through typed `packages/core` mutation ports; CLI may keep read-side lookup/orchestration, but it must not parse/stringify canonical frontmatter or assemble canonical write batches for those write paths.
3. Inbox capture persists raw evidence from iMessage, Telegram, Linq webhook chats, and AgentMail email connectors, appends typed `ledger/inbox-captures` facts plus compatibility event/audit projections, indexes attachments, and enqueues parse jobs in rebuildable local runtime state.
4. Parser workers or parsed-pipeline wrappers consume those attachment jobs and publish only derived artifacts.
5. Inbox model routing can materialize a normalized bundle, attach supported stored images for multimodal routing when available, and write audited bundle/plan/result artifacts before any optional apply step.
6. Importers may parse and normalize external inputs but must never write canonical vault files directly. Provider connectors normalize upstream payloads into shared device-batch payloads and still rely on `packages/core` for canonical persistence.
7. `packages/device-syncd` owns provider OAuth state, reconnect/disconnect control, scheduled device backfills, and optional webhook fan-in; its control routes must stay loopback-only plus bearer-authenticated, any public callback/webhook ingress should stay isolated from `/accounts/*` and `/providers/*`, polling-first providers remain first-class citizens, provider credentials stay outside the vault, per-account jobs should be serialized to avoid rotating-refresh-token races, and canonical health writes still flow through `packages/importers` and `packages/core`. Its provider-agnostic public OAuth/webhook handling should live in a reusable shared ingress layer so future hosted control planes and local tunnel setups do not fork provider callback logic. `packages/cli` may start, reuse, and stop that daemon for the active vault, but it should treat the localhost HTTP control plane as the stable boundary rather than reaching through to provider state in-process.
8. Provider-backed assistant chat and outbound channel flows may persist local session metadata, local transcript files, explicit delivery bindings, typed conversational memory docs, auto-reply cursors/channel state, scheduled assistant cron jobs, bounded non-canonical scratchpad state, and derived gateway conversation/message/event projections outside the vault, including selected non-canonical health context for continuity, but they must not treat that state as canonical health truth or bypass canonical write boundaries for health data. Saved assistant session bindings are monotonic routing facts: lookup may enrich missing channel, identity, participant, or thread fields, but it must fail closed rather than silently rebinding an existing session to a different audience; one-off outbound retargeting belongs on the explicit delivery-target override path instead of mutating the saved binding. The local `packages/assistantd` daemon is now an allowed loopback-only bearer-authenticated control boundary for this runtime, but it stays bound to one vault and does not become a second canonical write owner. Current outbound adapters include iMessage, Telegram, Linq, and AgentMail-backed email; email auto-reply is intentionally limited to direct threads and must preserve the inbound AgentMail inbox identity for replies, while Linq replies reuse the inbound chat id thread binding for the local webhook-driven conversation. Assistant inbox automation now merges canonical-routing candidates and reply candidates into one ordered decision pass so a capture is reasoned over once per scan even though reply and routing progress still keep separate internal cursors. New prompt-backed assistant cron jobs must declare an explicit outbound channel route and always deliver their generated response instead of storing local-only undelivered summaries, while optional `stateDocId` bindings let a cron carry forward scratch state only when configured. Provider turns now bind the real current user prompt, session id, and turn id on the host side, share one canonical local tool catalog across provider families, let Codex transports reach it through direct CLI/MCP wiring while OpenAI-compatible turns call it through the bound runtime, and keep prompt-only parity limited to the active vault rather than arbitrary host shell access. That shared catalog is still turn-scoped rather than universal: manual ask/deliver and assistant-cron turns may expose `assistant state list|show|put|patch|delete`, `assistant memory search|get|upsert|forget`, the `assistant cron` subtree, saved self-target lookups, and vault query/write tools, while assistant auto-reply is intentionally constrained to bounded `vault.fs.readText` inspection and bound-tool-capable provider routes only, so inbound prompt injection cannot fan out into broader local reads or writes via direct Codex CLI access. Provider execution context is rebuilt per failover route instead of reusing the primary-provider plan across provider families, assistant-memory commits stay serialized behind vault-scoped locks plus minimal provenance metadata, assistant runtime receipt/outbox/diagnostics/failover/status mutations stay serialized under one shared assistant-runtime write lock, and due cron jobs execute only while `vault-cli assistant run` is active for that vault.
9. Query/export paths are read-only and must not mutate canonical vault state.
10. The local web app reads vault data only on the server through query helpers, constrains search to safe record fields, and may redirect to the local authenticated device control plane for auth/account actions without gaining direct canonical write access.
11. The hosted `apps/web` control plane accepts only provider OAuth/webhook traffic plus authenticated browser/agent control traffic, keeps raw provider tokens away from browsers, stores only sparse Linq routing/event state instead of canonical captures, and should expose sparse sync hints/token export endpoints or queued Linq webhook events only to the paired local agent for the owning user. Browser-authenticated device-sync routes must rely on short-lived, request-bound signed assertions from a trusted auth edge or proxy, reject stale assertions, and consume browser-assertion nonces once so captured assertions cannot be replayed across hosted control-plane routes. Hosted onboarding routes separately rely on Privy-issued identity tokens from the `privy-id-token` cookie, keep SMS login and embedded-wallet creation browser-initiated through Privy's SDK user state, and centralize local server-side cookie verification with Privy's verification key before issuing a hosted session cookie. Cookie-authenticated hosted browser mutation routes must also enforce an allowed `Origin` check rather than relying on `SameSite` alone. Hosted internal device-sync runtime routes and hosted AI usage import routes are server-to-server only, authenticated by the hosted execution internal/control token, and must require a trusted worker-injected user binding that overrides or rejects any caller-supplied body `userId` or usage `memberId` so a compromised runner cannot override account scope. Hosted share payload reads now require the dedicated hosted share token plus that same trusted bound-user header instead of falling back to the broader hosted execution token. Hosted email ingress is its own trust boundary: resolving the recipient alias is not sufficient, so stable user aliases must only accept mail from the member's synced verified email and thread reply aliases must only accept that verified email or the saved thread participants before the worker stores the raw message or dispatches hosted execution. All Cloudflare-bound hosted execution from `apps/web`, including device-sync wake hints, hosted share acceptance imports, and hosted onboarding activation or Linq dispatches, must first be written to the shared Postgres `execution_outbox` in the same transaction as the originating state mutation; delivery to `apps/cloudflare` is then retried out of band, and hosted UI/API surfaces should read eventual execution state instead of depending on synchronous runner responses. Successful hosted assistant provider attempts now also land one immutable pending usage record under `assistant-state/usage/pending/`; after the hosted commit succeeds the runner best-effort imports those records into `apps/web` so hosted Postgres, not Stripe, remains the canonical hosted usage ledger. Hosted onboarding webhook receipts are now the retry journal only for receipt-local Linq or Telegram side effects; they durably persist the planned response plus queued side effects before any external send, keep a reclaimable processing lease, and mark hosted execution dispatch effects sent once the outbox row is durable rather than once Cloudflare transport returns inline. Stripe webhook ingress now verifies the event, normalizes the durable Stripe fact into Postgres, and returns quickly; a separate queued reconciler applies entitlement mutations, outbox dispatches, optional hosted AI token metering, and optional RevNet issuance out of band. Stripe billing state must remain monotonic under out-of-order delivery, including same-second collisions, so member entitlement writes must not rely on lexicographic Stripe event ids and stale success webhooks must not reactivate access after later cancellation or delinquency. Normal subscription delinquency, cancelation, and pause states block access through `billingStatus` without consuming the manual `member.status = suspended` lane, while refund/dispute/reversal outcomes still suspend hosted access, revoke live hosted sessions, and halt future activation or issuance. For subscription billing, `invoice.paid` is the only positive Stripe source and `customer.subscription.*` is negative/status-only; when optional RevNet issuance is configured, subscription activation moves to confirmed issuance for the paid invoice rather than raw `invoice.paid`. Once a broadcast tx hash exists, persistence failures fail closed for operator repair rather than classifying the attempt as a clean retry. This MVP still does not attempt onchain clawbacks.
12. The hosted `apps/cloudflare` execution plane accepts only signed internal dispatch at `POST /internal/dispatch` or operator-authenticated internal control calls at `GET /internal/users/:userId/status`, `POST /internal/users/:userId/run`, `GET|PUT|DELETE /internal/users/:userId/env`, and the internal gateway control routes under `POST /internal/users/:userId/gateway/**`, fails those worker/container control paths closed when `HOSTED_EXECUTION_CONTROL_TOKEN` or `HOSTED_EXECUTION_RUNNER_CONTROL_TOKEN` are missing, routes worker control calls into direct Durable Object methods instead of worker-local callback fetches, restores an ephemeral local execution context from encrypted hosted bundle snapshots, performs durable hosted bootstrap explicitly during `member.activated` rather than on every event run, injects per-user runner env from a separate encrypted hosted object at runner start, starts the per-user Durable Object's native container to run the one-shot runner, and keeps the hosted gateway read model as a hot derived projection plus short-retained event log owned by that Durable Object rather than by `apps/web` or the runner workspace itself. Hosted prompt-backed assistant automation now runs by default; `HOSTED_EXECUTION_ENABLE_ASSISTANT_AUTOMATION=false` is the explicit opt-out when the hosted runner should stay on deterministic inbox/parser/device-sync/share-import work, avoid auto-enabling assistant reply channels, and strip the broader automation-only secret set from the runner. The outer container wrapper now authenticates `/internal/invoke` and `/internal/destroy` with the runner control token header before it trusts any JSON body, while the runner reaches back to the hosted web control plane only through worker-owned outbound proxy hosts for device-sync hydration/reconciliation, share payload reads, and hosted AI usage imports. Those proxy hosts keep the broad web control token inside the worker, inject the Durable Object's bound `userId`, and preserve the fail-closed trust boundary even if one runner container is compromised. The isolated hosted child no longer inherits the full container supervisor env or its ambient proxy credentials, now uses per-run writable cache and temp roots while avoiding the supervisor `HOME` at launch, and is killed as a process group when the container request aborts so warm-container reuse does not implicitly preserve the previous run's writable state. Callback and control-plane base URLs must normalize to HTTPS unless they are explicit loopback or worker-host allowlist exceptions. The per-user queue keeps queue/poison tracking in Durable Object SQLite tables and blocks replayed immutable event ids with 30-day exact tombstones rather than an indefinite Bloom-style filter, writes updated encrypted bundles back to object storage, skips bundle rewrites when the content hash is unchanged, centralizes the default callback hosts plus canonical dispatch lifecycle and outbox payload helpers in `packages/hosted-execution`, decrypts stored bundle/artifact/journal objects by envelope `keyId` through the configured keyring, commits only the final bundle-ref/version transition inside Durable Object storage after R2 I/O succeeds, and schedules the next per-user wake without inventing a second canonical persistence model. Hosted runner observability now stays in that same Durable Object lane: `runner_meta` persists the latest run summary plus a bounded recent timeline keyed by the shared `runId` / `attempt` / `phase` contract, while worker, container, and runtime boundaries emit matching structured logs so Cloudflare remains the searchable debug surface instead of a second source of truth. Those logs intentionally default to redacted operator-safe summaries and do not auto-attach the bound `userId`; callers must opt in explicitly if a user identifier is truly required.

## CLI Framework Notes

- `packages/cli` is built on incur. Model nested verbs with real mounted sub-CLIs such as `search -> query` and `search -> index -> status|rebuild`; do not simulate nested commands with argv rewrites or positional action enums.
- Treat output/discovery transport such as `--format`, `--json`, `--verbose`, `--schema`, `--llms`, `skills add`, and `--mcp` as incur-owned global behavior. Murph command docs should focus on domain semantics unless the repo intentionally constrains that surface.
- Keep the root CLI default-exported from `packages/cli/src/index.ts` and keep `packages/cli/src/incur.generated.ts` aligned with command-topology changes so typed CTAs and generated skill metadata stay truthful.
- Source-only CLI checks are useful for triage, but repo acceptance still depends on the built CLI path because package tests execute `packages/cli/dist/bin.js`.

## Current Verification Posture

The repository still uses the bootstrap verification commands, but it now also has a repo-owned parser bootstrap path (`pnpm setup:inbox`), a cross-platform repo-local host bootstrap path (`pnpm onboard` / `scripts/setup-host.sh`) for macOS and Linux, a fixed-version monorepo release manifest that packs and publishes the direct CLI runtime package chain under one tag, a local web package that builds under Next.js' default Turbopack path with source-aware workspace resolution, a local device-sync runtime with service/http tests, and inbox/parser package tests that exercise runtime rebuild, parser workers, parser-toolchain discovery, and parsed-pipeline flows inside the local TypeScript workspace.
