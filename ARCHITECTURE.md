# Murph Architecture

Last verified: 2026-04-04

## Module Map

- `packages/contracts`: canonical Zod contracts, parse helpers, TypeScript types, and generated JSON Schema artifacts
- `packages/hosted-execution`: shared hosted control-plane contracts, HMAC signing/verification helpers, env readers, route builders, side-effect codecs, and typed clients for the hosted web and Cloudflare execution apps
- `packages/runtime-state`: shared hosted email/env/loopback/id helpers plus pure hosted bundle identity types/equality on the root package, with an explicit `@murphai/runtime-state/node` subpath for hosted bundle codec/materialization, `.runtime` path resolution, rebuildable local-state helpers, assistant-state files, process scoping, and SQLite-backed Node-only runtime seams
- `packages/core`: the only package allowed to mutate canonical vault data
- `packages/importers`: ingestion adapters that parse external files or provider API snapshots, normalize them behind registry-based adapters, and delegate all writes to core
- `packages/device-syncd`: published local device OAuth/webhook/reconcile runtime with an authenticated localhost control plane, optional separate public callback/webhook ingress, a reusable shared public-ingress core for future hosted/tunneled callback surfaces, the canonical `@murphai/device-syncd/client` control-plane client/contracts surface for non-daemon callers, encrypted provider credentials outside the vault, and normalized provider snapshot imports through importers/core
- `packages/messaging-ingress`: shared stateless messaging-provider ingress package that owns provider webhook parsing/verification, target grammar, supported-message extraction, summary helpers, and sparse raw minimization for transports such as Telegram and Linq without taking on polling drivers, hosted policy, or runtime persistence
- `packages/inboxd`: inbox capture ingestion/runtime package that persists canonical raw inbox evidence plus a first-class append-only inbox-capture ledger while keeping inbox-only cursors, source-specific checkpoints, capture indexes, and attachment job state in local SQLite state; it consumes `@murphai/messaging-ingress` for stateless Telegram/Linq ingress semantics while continuing to own polling connectors, local capture persistence, and the optional inbox-plus-parser daemon composition helpers layered on top of parser-owned runtime contracts
- `packages/parsers`: local-first attachment parsing, parser-service helpers, parser-owned runtime/store contracts for attachment parsing, and derived artifact publication under `derived/inbox/**`; it no longer owns inbox daemon orchestration or depends upward on `@murphai/inboxd`
- `packages/query`: read helpers, export-pack generation, the semantic wearable read model and day-summary layer over imported device evidence, the optional lexical search index over canonical vault data, and the pure parser/search/index helpers for derived knowledge pages under `derived/knowledge/**`
- `packages/gateway-core`: published transport-neutral gateway boundary package that owns the shared gateway contracts, route helpers, projection/snapshot logic, opaque ids, and event-log helpers used by both local and hosted adapters
- `packages/gateway-local`: published local vault-backed gateway runtime package that owns the rebuildable `.runtime/gateway.sqlite` serving store plus the local gateway read/send/permission wrapper surface used by assistantd and restored hosted-runtime workspaces; it depends only on lower-layer gateway contracts and receives assistant-backed projection/send adapters by injection from higher layers
- `packages/assistantd`: published local assistant daemon package with a bearer-authenticated loopback-only control plane bound to one vault; it fronts steady-state local assistant session/message/status/automation entrypoints through the explicit `@murphai/assistant-core` headless surface and now hosts the full local gateway conversation/message/attachment/send/event/permission API over `@murphai/gateway-local`, serving reads from the rebuildable local gateway serving store without moving canonical writes out of the CLI/core boundary
- `packages/assistant-runtime`: headless hosted assistant execution surface that exposes one-shot inbox/bootstrap/assistant/outbox/device-sync/share-import runtime behavior behind explicit runtime context, consumes the same `@murphai/assistant-core` headless boundary directly instead of a second wrapper package, now treats the durable operator `hostedAssistant` config as the only persisted hosted assistant source of truth, and consumes shared messaging ingress contracts from `@murphai/messaging-ingress` rather than defining provider semantics itself
- `apps/web`: hosted Next.js integration control plane for Vercel-style deployments, backed by Postgres/Prisma for device OAuth sessions, encrypted token escrow, ownership mapping, webhook traces, sparse sync signals, internal snapshot/apply routes for hosted runner hydration, a durable `execution_outbox` for Cloudflare-bound hosted execution intents, immutable hosted AI usage rows imported after successful hosted commits, Linq recipient bindings, queued Linq webhook events, per-member hosted Linq daily message state, queued hosted Stripe event facts, hosted onboarding webhook receipts for receipt-local side effects, Privy-backed hosted onboarding state, hosted checkout attempts, optional invoice-idempotent RevNet issuance records, one-time encrypted hosted share links, and local-agent pairing while keeping canonical health and inbox data out of the hosted tier; the app layer keeps webhook trust/member/privacy policy while consuming stateless provider ingress helpers from `@murphai/messaging-ingress`
- `apps/cloudflare`: hosted execution plane for signed internal dispatch, per-user coordination via container-enabled Durable Objects, encrypted hosted workspace snapshot storage plus separately encrypted artifact objects, a native Cloudflare container image, an encrypted hot per-user gateway projection plus short-retained gateway event log for internal control-plane reads, and one-shot inbox/parser/assistant/device-sync/share-import execution that materializes a temporary local workspace through `packages/assistant-runtime` rather than importing hosted seams from the operator CLI surface
- `packages/cli`: the published `@murphai/murph` package plus the `murph` and `vault-cli` binaries, an incur-backed typed operator surface over core/importers/query/inboxd plus parser-toolchain queue controls, quick workout capture atop canonical `activity_session` events, inbox model-routing helpers, provider-backed assistant session orchestration, the transport-neutral `@murphai/gateway-core` headless conversation/message/event/send contract now shared directly by assistantd and hosted Cloudflare gateway adapters (with MCP still intentionally deferred), a local gateway projection store under `.runtime/gateway.sqlite` now owned by `@murphai/gateway-local`, incrementally syncing inbox captures through inboxd's durable local capture mutation cursor while still syncing assistant sessions/outbox via their existing signatures, persisting explicit snapshot bootstrap metadata so empty projections do not rebuild forever, and serving conversation/message/attachment reads plus event polling from the rebuilt store, optional env-routed client access to `packages/assistantd` for steady-state local assistant open/send/update flows plus session/status/outbox/cron inspection and serializable automation control, CLI-owned command/UI/client helpers for the selected vault, out-of-vault assistant memory docs plus one shared bound assistant/vault tool catalog with turn-scoped profiles for assistant memory/state/cron/knowledge, saved self-target inspection, explicitly enabled guarded public `web.fetch` and `web.pdf.read`, provider-agnostic `web.search` when a search backend is configured, bounded vault text reads, and vault query/write operations across provider turns, outbound iMessage/Telegram/Linq/AgentMail email channel adapters, an onboarding/setup wizard that can reuse or discover existing AgentMail inboxes before provisioning, and local host setup commands for macOS and Linux while keeping iMessage itself macOS-only. The dedicated `@murphai/assistant-core` package owns the local-only assistant/inbox/vault/operator-config surface plus the shared `usecases/**`, `inbox-app/**`, setup/runtime-helper layers, the shared derived-knowledge service, and the assistant-native `assistant.knowledge.*` tool surface for non-CLI consumers and for CLI headless imports; `packages/cli` now imports that owner package directly and keeps only the CLI command/setup glue and daemon-aware wrapper modules that genuinely live at the CLI transport boundary.
- `fixtures/` and `e2e/`: deterministic fixture corpus and end-to-end smoke flows

## Trust Boundaries

- Canonical vault storage is file-native under the vault root.
- Human-facing truth lives in Markdown documents such as `CORE.md`, journal pages, and experiment pages.
- Machine-facing truth lives in append-only JSONL ledgers for events, samples, and audit records.
- Raw imported artifacts are immutable once copied into `raw/`, including normalized device/provider API snapshots under `raw/integrations/**`.
- Parser outputs under `derived/inbox/**` are rebuildable and never canonical health facts.
- Model-authored compiled knowledge pages under `derived/knowledge/**` are non-canonical, rebuildable dossiers that synthesize local vault evidence and saved research notes without becoming a second source of truth.
- Inbox model-routing bundles, plans, and execution results under `derived/inbox/**/assistant/*.json` are rebuildable audit artifacts and never canonical health facts.
- Inbox runtime state is local-only under `.runtime/inboxd.sqlite` plus `.runtime/inboxd/*.json` and is rebuildable from canonical vault evidence under `raw/inbox/**`.
- AgentMail inbox ids, outbound email thread bindings, and channel credentials remain non-canonical local/runtime concerns; AgentMail API keys stay in operator environment variables and never belong in the vault.
- Query search runtime state is local-only under `.runtime/search.sqlite` and is rebuildable from canonical vault evidence.
- Device sync runtime state is local-only under `.runtime/device-syncd.sqlite`, and Murph's daemon launcher state/logs live under `.runtime/device-syncd/`; encrypted provider tokens, OAuth sessions, and webhook/reconcile cursors never belong in the canonical vault. Hosted device-sync control-plane state for `apps/web` belongs in Postgres only and must remain limited to integration/auth state rather than canonical health payloads. Hosted onboarding billing rows, checkout attempts, queued Stripe facts, webhook receipts, the shared `execution_outbox`, immutable hosted AI usage rows, and optional RevNet issuance rows in Postgres are operational/idempotency state only, not canonical health truth.
- Local assistant runtime state is non-canonical under sibling `assistant-state/**`, including sessions, transcripts, outbox/receipt artifacts, diagnostics, status, and memory/state documents. Murph may replay recent raw transcript turns and rely on provider-native resume or compaction for continuity, but that runtime context still must never be treated as canonical health memory or vault truth.
- The gateway plane is a derived operational model over inbox captures, assistant bindings, sent outbox deliveries, and approval state. Local daemon APIs and hosted Durable Objects may materialize hot gateway projections and short-retained event logs for transport-facing reads, but those projections are never canonical health truth and must remain rebuildable from canonical vault evidence plus non-canonical runtime state. The local projection now persists serving tables in `.runtime/gateway.sqlite`, is owned by `@murphai/gateway-local`, consumes inboxd's durable local capture mutation cursor for incremental capture sync, persists explicit snapshot bootstrap metadata so empty projections stay incremental instead of rebuilding forever, and keeps full rebuild behavior as the bootstrap or recovery path.
- Hosted execution state for `apps/cloudflare` may store one encrypted hosted workspace snapshot in the existing `vault` bundle slot plus separately encrypted artifact objects and per-user coordination metadata. That hosted workspace snapshot now includes canonical `vault/**`, durable `vault/.runtime/**`, sibling `assistant-state/**`, and the minimal operator-home config needed for explicit `member.activated` bootstrap, including the durable top-level `hostedAssistant` operator config while keeping raw provider credentials in Worker secrets or the separately encrypted per-user env object. Large raw artifacts under `vault/raw/**` are externalized into their own encrypted objects. The runner restores inline workspace files first, materializes only the externalized artifact paths the current run actually needs, preserves untouched artifact refs across later snapshots so old media is not re-downloaded just to survive the next commit, and best-effort deletes only per-user artifact objects no longer referenced by the latest workspace snapshot. Hosted per-user env overrides still live in their own encrypted object keyed independently from the workspace snapshot. Those hosted bundles still represent canonical vault material and must be treated as sensitive data rather than a separate hosted source of truth. The worker-facing control routes are internal only, the per-user Durable Object now keeps its queue/process state in SQLite tables plus encrypted bundle refs rather than one serialized record blob, starts its native Cloudflare container and talks to the runner over the container TCP port, commits only small bundle-ref/version state changes inside Durable Object storage after any R2 I/O completes, persists committed execution journal records with the owning `userId` so recovery can still rebind the Durable Object after partial state loss, and also maintains an encrypted hot per-user gateway projection snapshot plus short-retained gateway event log derived from commit/finalize callbacks rather than from a second hosted source of truth. The containerized runner now hydrates hosted device-sync runtime state from `apps/web`, resolves hosted share payload reads, hosted wearable connect-link creation, and hosted AI usage imports only through worker-owned outbound proxy hosts that inject the bound `userId`, keeps the broad web internal/control token out of the runner environment, launches each hosted job in a scrubbed child env with per-run temp and cache roots instead of inheriting the full supervisor env, avoids the supervisor `HOME` at child launch while still restoring the persisted operator-home snapshot inside the workspace for runtime config, strips ambient supervisor proxy credentials from the isolated child env, reaps the whole child process group when the container request times out or disconnects, persists due hosted side effects on the committed journal, authenticates the outer container invoke/destroy wrapper with the runner control token header before any body is trusted, accepts hosted gateway send requests only through the same route-bound gateway seam, and calls the worker's internal commit/finalize/side-effect routes so post-commit retries can resume outward delivery without rerunning compute. Exact consumed-event tombstones now block replayed hosted dispatch event ids for 30 days; there is no secondary replay filter beyond that authoritative window. Hosted ciphertext envelopes still decrypt by envelope `keyId` through the configured keyring so operators can stage bundle-key rotation without eagerly rewriting older objects. Writes still use one active bundle key at a time, and missing keyring entries still fail closed. This lane does not claim operator-blind privacy or TEE isolation.
- Any inbox-to-canonical promotion idempotency must be stored in or derivable from canonical vault evidence, not `.runtime/` alone.
- General assistant/session state belongs outside the canonical vault under `assistant-state/`, including local transcript files, assistant cron scheduler state and run history under `assistant-state/cron/`, per-turn decision receipts under `assistant-state/receipts/`, replay-safe outbound intent journals under `assistant-state/outbox/`, pending provider-usage records under `assistant-state/usage/pending/`, diagnostics events plus snapshot counters under `assistant-state/diagnostics/`, persisted assistant status snapshots under `assistant-state/status.json`, provider failover cooldown state under `assistant-state/failover.json`, generic small JSON scratchpad documents under `assistant-state/state/`, plus non-canonical Markdown memory for naming, response preferences, standing instructions, selected health context, and recent project context; capture-scoped rebuildable audit artifacts stay under `derived/inbox/**`, while durable compiled knowledge dossiers live under `derived/knowledge/**`.
- Provider transcript history and channel-native delivery history should stay with upstream adapters when possible; Murph stores local assistant transcript copies, minimal manual aliases, explicit conversation bindings, automation cursors, enabled auto-reply channel state, timestamps/turn counts, provider session references, cron job definitions and run history, compact system-emitted turn receipts, idempotent outbound intent state, diagnostics counters/warnings, persisted failover cooldown metadata, persisted status snapshots, generic assistant scratchpad documents, and Markdown assistant memory under `assistant-state/`. Assistant-state directories must stay private (`0700`) and assistant-state files must stay private (`0600`). Secret-bearing provider headers for persisted sessions live only in private sidecars under `assistant-state/secrets/**`; the general session JSON keeps only public headers, diagnostics/runtime-event writes redact inline secret material before persistence, and `assistant doctor --repair` can tighten permissive assistant-state modes in place. Inline secret findings indicate stale local session data rather than a supported migration path. Fresh sessions inject only a small core block by default, recent/project memory is retrieved on demand through typed `assistant memory search|get|forget` calls, and provider turns now use one shared CLI-first Murph runtime surface plus a small helper-tool layer across manual, cron, and message-triggered automation turns. OpenAI-compatible routes reach canonical Murph operations through the shared `murph.cli.run` executor plus a few bounded helper tools, while Codex remains a privileged local adapter that can call the same `vault-cli` surface through its native local CLI/filesystem/env authority. Assistant memory Markdown file reads and edits are now part of that canonical CLI surface through `assistant memory file read|append|write` rather than separate provider-only memory-edit tools. Optional provider-agnostic `web.search` plus explicitly opt-in `web.fetch` and `web.pdf.read` stay available as public read-only external-context tools rather than canonical data paths. These web-read tools remain HTTP(S) read surfaces only: they strip URL fragments, redact query-bearing URL details from returned metadata and errors, resolve and pin vetted public destination addresses before each request or redirect so the socket does not silently re-resolve to a different host, block private-network destinations, bound response and PDF-parse work, and return untrusted external content rather than canonical Murph data. Accepted inbound channel messages are therefore treated as operator-authorized actions for the bound vault and may use the full assistant-state, assistant-memory, assistant-cron, self-target, and vault query/write surface. Murph owns transcript policy, turn orchestration, and tool/runtime planning across providers, but Codex CLI is now an explicitly privileged local adapter rather than a boxed-in host-tool client: it keeps native local CLI/filesystem/env authority, defaults to unsandboxed execution plus no approval friction, and is trusted as a local operator path. Generic scratchpads are exposed through `assistant state list|show|put|patch|delete`, assistant-written memory is committed only through host-bound writes that attach session/turn provenance, and canonical vault records remain authoritative on conflicts.
- `vault-cli inbox model route` may send a normalized inbox bundle, with raw routing images attached when eligible, to either the AI Gateway or an operator-specified OpenAI-compatible endpoint.

## Control Flow

1. Operators, automations, and future agent layers call `vault-cli` or package APIs.
2. CLI commands stay thin, validate input, and delegate to internal CLI use-case modules that coordinate `packages/core`, `packages/importers`, `packages/query`, `packages/inboxd`, and parser-toolchain helpers from `packages/parsers`. Canonical mutation flows for experiments, journal pages, providers, events, vault summary updates, and inbox journal/experiment-note promotions must route through typed `packages/core` mutation ports; CLI may keep read-side lookup/orchestration, but it must not parse/stringify canonical frontmatter or assemble canonical write batches for those write paths.
3. Inbox capture persists raw evidence from iMessage, Telegram, Linq webhook chats, and AgentMail email connectors, appends typed `ledger/inbox-captures` facts plus compatibility event/audit projections, indexes attachments, and enqueues parse jobs in rebuildable local runtime state.
4. Parser workers or parsed-pipeline wrappers consume those attachment jobs and publish only derived artifacts.
5. Inbox model routing can materialize a normalized bundle, attach supported stored images for multimodal routing when available, and write audited bundle/plan/result artifacts before any optional apply step.
6. Importers may parse and normalize external inputs but must never write canonical vault files directly. Provider connectors normalize upstream payloads into shared device-batch payloads and still rely on `packages/core` for canonical persistence.
7. `packages/device-syncd` owns provider OAuth state, reconnect/disconnect control, scheduled device backfills, and optional webhook fan-in; its control routes must stay loopback-only plus bearer-authenticated, any public callback/webhook ingress should stay isolated from `/accounts/*` and `/providers/*`, polling-first providers remain first-class citizens, provider credentials stay outside the vault, per-account jobs should be serialized to avoid rotating-refresh-token races, and canonical health writes still flow through `packages/importers` and `packages/core`. Its provider-agnostic public OAuth/webhook handling should live in a reusable shared ingress layer so future hosted control planes and local tunnel setups do not fork provider callback logic. `packages/cli` may start, reuse, and stop that daemon for the active vault, but it should treat the localhost HTTP control plane as the stable boundary rather than reaching through to provider state in-process.
8. Provider-backed assistant chat and outbound channel flows may persist local session metadata, local transcript files, explicit delivery bindings, typed conversational memory docs, auto-reply cursors/channel state, scheduled assistant cron jobs, bounded non-canonical scratchpad state, and derived gateway conversation/message/event projections outside the vault, including selected non-canonical health context for continuity, but they must not treat that state as canonical health truth or bypass canonical write boundaries for health data. Saved assistant session bindings are monotonic routing facts: lookup may enrich missing channel, identity, participant, or thread fields, but it must fail closed rather than silently rebinding an existing session to a different audience; one-off outbound retargeting belongs on the explicit delivery-target override path instead of mutating the saved binding. The local `packages/assistantd` daemon is now an allowed loopback-only bearer-authenticated control boundary for this runtime, but it stays bound to one vault and does not become a second canonical write owner. Current outbound adapters include iMessage, Telegram, Linq, and AgentMail-backed email; email auto-reply is intentionally limited to direct threads and must preserve the inbound AgentMail inbox identity for replies, while Linq replies reuse the inbound chat id thread binding for the local webhook-driven conversation. Assistant inbox automation now merges canonical-routing candidates and reply candidates into one ordered decision pass so a capture is reasoned over once per scan even though reply and routing progress still keep separate internal cursors. New prompt-backed assistant cron jobs must declare an explicit outbound channel route and always deliver their generated response instead of storing local-only undelivered summaries, while optional `stateDocId` bindings let a cron carry forward scratch state only when configured. Provider turns now bind the real current user prompt, session id, and turn id on the host side, share one CLI-first Murph runtime surface across provider families, and keep provider differences focused on transport/runtime behavior rather than Murph authority. OpenAI-compatible routes call the bound runtime through the shared `murph.cli.run` executor plus bounded helper tools, while Codex reaches the same canonical `vault-cli` surface through its native local CLI authority; memory Markdown file edits now go through the same CLI path via `assistant memory file ...` instead of direct provider-only tools. Message-triggered assistant turns now use that same full Murph runtime surface rather than a bounded read-only profile, so any accepted inbound channel message can inspect and mutate assistant memory/state/cron plus canonical vault records for the bound user and vault. The provider still changes the transport mechanism for those actions, but not whether the bound user/vault authority exists. Provider execution context is rebuilt per failover route instead of reusing the primary-provider plan across provider families, assistant-memory commits stay serialized behind vault-scoped locks plus minimal provenance metadata, assistant runtime receipt/outbox/diagnostics/failover/status mutations stay serialized under one shared assistant-runtime write lock, and due cron jobs execute only while `vault-cli assistant run` is active for that vault.
9. Query/export paths are read-only and must not mutate canonical vault state.
11. The hosted `apps/web` control plane accepts only provider OAuth/webhook traffic plus authenticated browser/agent control traffic, keeps raw provider tokens away from browsers, stores only sparse Linq routing/event state instead of canonical captures, purpose-derives separate hosted escrow cipher scopes for access versus refresh tokens, records durable audit rows for token export/refresh actions, and should expose sparse sync hints/token export endpoints or queued Linq webhook events only to the paired local agent for the owning user. Hosted Linq onboarding now treats any valid first-contact inbound as an onboarding request: it creates or reuses the hosted member keyed by the blinded phone lookup, persists the Linq chat binding on the member for post-payment follow-up, sends at most one onboarding link per UTC day while the member is still inactive, counts echoed outbound Linq webhooks against the same per-member UTC-day state, and enforces a 100 inbound-message UTC-day cap for billing-active members before writing the hosted execution dispatch outbox row, with at most one quota reply per UTC day. The only human/browser wearable-management surface is `/api/settings/device-sync/**`; it relies on Privy-issued identity tokens from the `privy-id-token` cookie plus allowed `Origin` checks, keeps SMS login and embedded-wallet creation browser-initiated through Privy's SDK user state, and centralizes local server-side cookie verification with Privy's verification key before issuing a hosted session cookie. Lower-level assertion-authenticated browser bridge routes such as `POST /api/device-sync/agents/pair` must rely on short-lived, request-bound signed assertions from a trusted auth edge or proxy, reject stale assertions, and consume browser-assertion nonces once so captured assertions cannot be replayed across hosted control-plane routes. Remaining hosted web internal routes are server-to-server only, still authenticated by the hosted execution internal/share bearer tokens plus trusted bound-user headers where applicable, and must not trust caller-supplied member scoping over a trusted scheduler or server binding. The runner hot path no longer reaches back into hosted web for device-sync runtime snapshots, share payload reads, or direct usage-record writes; `apps/web` now publishes those runtime inputs into Cloudflare-owned mirrors and later imports pending hosted usage by reading Cloudflare-owned storage. Hosted assistant wearable connect-link creation is the narrow exception: the runner still reaches `apps/web` only through the Cloudflare-owned `device-sync.worker` proxy host, and the worker adds the bound user header plus internal bearer token so the runner never sees broad hosted-web credentials. Hosted email ingress is its own trust boundary: resolving the recipient alias is not sufficient, so the worker must re-authorize the sender before raw-message persistence or hosted dispatch. New outbound mail keeps the fixed public sender identity but uses one stable per-user reply alias, registered members may also initiate a conversation by emailing that fixed public sender address once their verified email has been synced into hosted execution, direct-public-mail lookup is allowed only through an encrypted verified-owner index that stores secret-derived sender hashes instead of raw verified emails, that direct-public path now requires matching envelope and `From` sender values before lookup, unresolved or unauthorized mail to the public sender address is accepted-and-dropped to reduce account-state probing, leaked aliases are never sufficient authority on their own, and hosted email ingress no longer accepts legacy per-thread reply aliases or legacy verified-owner route records. All Cloudflare-bound hosted execution from `apps/web`, including device-sync wake hints, hosted share acceptance imports, and hosted onboarding activation or Linq dispatches, must first be written to the shared Postgres `execution_outbox` in the same transaction as the originating state mutation; delivery to `apps/cloudflare` is then retried out of band, and hosted UI/API surfaces should read eventual execution state instead of depending on synchronous runner responses. Successful hosted assistant provider attempts now also land one immutable pending usage record under `assistant-state/usage/pending/`; `apps/web` later imports those records from the Cloudflare-owned pending-usage store so hosted Postgres, not Stripe, remains the canonical hosted usage ledger. Hosted onboarding webhook receipts are now the retry journal only for receipt-local Linq or Telegram side effects; they durably persist the planned response plus queued side effects before any external send, keep a reclaimable processing lease, and mark hosted execution dispatch effects sent once the outbox row is durable rather than once Cloudflare transport returns inline. Stripe webhook ingress now verifies the event, stores a durable Stripe fact row, immediately tries to reconcile that specific event inline, then best-effort drains the matching hosted execution outbox rows; the Stripe recovery cron handles failed or deferred Stripe facts, optional hosted AI token metering, and optional RevNet issuance follow-up out of band. Member activation dispatches can now carry a minimal encrypted first-contact delivery locator so the hosted runtime, not `apps/web`, sends the deterministic Murph welcome as a real assistant-originated first turn in the existing conversation thread. Stripe billing state must remain monotonic under out-of-order delivery, including same-second collisions, so member entitlement writes must not rely on lexicographic Stripe event ids and stale success webhooks must not reactivate access after later cancellation or delinquency. Normal subscription delinquency, cancelation, and pause states block access through `billingStatus` without consuming the manual `member.status = suspended` lane, while refund/dispute/reversal outcomes still suspend hosted access, revoke live hosted sessions, and halt future activation or issuance. For subscription billing, `invoice.paid` is the only positive Stripe source and `customer.subscription.*` is negative/status-only; when optional RevNet issuance is configured, subscription activation moves to confirmed issuance for the paid invoice rather than raw `invoice.paid`. Once a broadcast tx hash exists, persistence failures fail closed for operator repair rather than classifying the attempt as a clean retry. This MVP still does not attempt onchain clawbacks.
12. The hosted `apps/cloudflare` execution plane accepts only signed internal dispatch at `POST /internal/dispatch` or HMAC-signed internal control calls at `GET /internal/users/:userId/status`, `POST /internal/users/:userId/run`, `GET|PUT|DELETE /internal/users/:userId/env`, `GET|PUT /internal/users/:userId/keys/envelope`, `PUT /internal/users/:userId/keys/recipients/:kind`, `PUT /internal/users/:userId/device-sync/runtime-snapshot`, `PUT /internal/users/:userId/shares/:shareId/pack`, `GET|DELETE /internal/users/:userId/usage/pending`, and the internal gateway control routes under `POST /internal/users/:userId/gateway/**`. It routes worker control calls into direct Durable Object methods instead of worker-local callback fetches, restores an ephemeral local execution context from encrypted hosted bundle snapshots, performs canonical hosted workspace bootstrap explicitly during `member.activated`, persists or adopts one explicit durable hosted assistant profile in operator config, injects per-user runner env from a separate encrypted hosted object at runner start, starts the per-user Durable Object's native container to run the one-shot runner, and keeps the hosted gateway read model plus device-sync runtime mirrors, share packs, pending usage buffers, dispatch payload refs, and per-user key envelopes in Cloudflare-owned encrypted storage rather than in `apps/web`. Hosted prompt-backed assistant automation runs whenever that explicit hosted assistant profile is present and ready; missing or invalid hosted assistant config fails closed by skipping assistant automation and hosted auto-reply channel enablement instead of falling back implicitly. The outer container wrapper authenticates each run with a one-shot injected runner control token before it trusts any JSON body, while the runner reaches back through worker-owned outbound proxy hosts only for worker-owned device-sync/share/usage state plus commit-side-effect callbacks. Those proxy hosts keep the runner untrusted, inject the Durable Object's bound `userId`, and preserve the fail-closed trust boundary even if one runner container is compromised. The isolated hosted child no longer inherits the full container supervisor env or its ambient proxy credentials, now uses per-run writable cache and temp roots while avoiding the supervisor `HOME` at launch, and is killed as a process group when the container request aborts so warm-container reuse does not implicitly preserve the previous run's writable state. Callback and control-plane base URLs must normalize to HTTPS unless they are explicit loopback or worker-host allowlist exceptions. The per-user queue keeps queue/poison tracking in Durable Object SQLite tables and blocks replayed immutable event ids with 30-day exact tombstones rather than an indefinite Bloom-style filter, persists only inline/reference dispatch envelopes in SQLite while externalizing sensitive Linq, Telegram, gateway-send, and device-sync payload bodies into separately encrypted transient blobs, writes updated encrypted bundles back to object storage through opaque bundle/artifact/env/journal locators, skips bundle rewrites when the content hash is unchanged, centralizes the default callback hosts plus canonical dispatch lifecycle and outbox payload helpers in `packages/hosted-execution`, decrypts stored bundle/artifact/journal objects by envelope `keyId` through the configured keyring, best-effort deletes consumed transient queue payload blobs and hosted raw email bodies after successful finalize while short R2 lifecycle rules backstop any leftovers, commits only the final bundle-ref/version transition inside Durable Object storage after R2 I/O succeeds, and schedules the next per-user wake without inventing a second canonical persistence model. Hosted runner observability now stays in that same Durable Object lane: `runner_meta` persists the latest run summary plus a bounded recent timeline keyed by the shared `runId` / `attempt` / `phase` contract, while worker, container, and runtime boundaries emit matching structured logs so Cloudflare remains the searchable debug surface instead of a second source of truth. Those logs intentionally default to redacted operator-safe summaries and do not auto-attach the bound `userId`; callers must opt in explicitly if a user identifier is truly required.

## CLI Framework Notes

- `packages/cli` is built on incur. Model nested verbs with real mounted sub-CLIs such as `search -> query` and `search -> index -> status|rebuild`; do not simulate nested commands with argv rewrites or positional action enums.
- Treat output/discovery transport such as `--format`, `--json`, `--verbose`, `--schema`, `--llms`, `skills add`, and `--mcp` as incur-owned global behavior. Murph command docs should focus on domain semantics unless the repo intentionally constrains that surface.
- Keep the root CLI default-exported from `packages/cli/src/index.ts` and keep `packages/cli/src/incur.generated.ts` aligned with command-topology changes so typed CTAs and generated skill metadata stay truthful.
- Source-only CLI checks are useful for triage, but repo acceptance still depends on the built CLI path because package tests execute `packages/cli/dist/bin.js`.
